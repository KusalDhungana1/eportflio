Full Stack Development

Architecture, APIs, Auth & Workflows

Full Stack Development is the practice of working across all layers of a web application — from the pixels a user sees to the database rows that store their data. A full-stack developer understands how the Frontend (client-side UI built with HTML, CSS, JavaScript), the Backend (server-side logic using Node.js, Python, PHP, etc.), and the Database (SQL or NoSQL storage) interconnect to deliver a seamless product.

The three layers are often called the Three-Tier Architecture: the Presentation Tier renders the UI; the Logic Tier processes business rules; and the Data Tier persists and retrieves information. Each tier communicates over standardised protocols, most commonly HTTP/HTTPS using REST or GraphQL APIs.

Three-Tier System Architecture

Every request travels through the full stack: the browser sends an HTTP request → the server validates and processes it → queries the database → sends JSON back → the browser re-renders the UI.

[ Client / Browser ]   — HTML · CSS · JS — renders the UI
        │ HTTP Request (GET /api/users)
        ▼
[ Server / Node.js + Express ]   — routes, middleware, controllers
        │ SQL / NoSQL Query
        ▼
[ Database / PostgreSQL / MongoDB ]   — stores & retrieves data
        │ Result Set / Documents
        ▲
[ Server ]   — serialises to JSON, applies CORS headers
        │ HTTP Response 200 OK + JSON body
        ▲
[ Client ]   — parses JSON, updates DOM

Authentication & Authorisation

Authentication answers "Who are you?" — verifying identity. Authorisation answers "What can you do?" — enforcing permissions. These are distinct concerns that must never be conflated.

The industry standard is JWT (JSON Web Token): a signed, URL-safe token containing a header, payload (claims), and signature. The server issues the JWT on login; subsequent requests include it in the Authorization: Bearer <token> header. The server validates the signature without touching the database, enabling stateless authentication. 

// 1. Login — server returns JWT
POST /api/auth/login  →  { "token": "eyJhbGci..." }

// 2. Protected request — client sends JWT
GET /api/profile
Authorization: Bearer eyJhbGci...

// 3. Server verifies signature, extracts userId
jwt.verify(token, SECRET_KEY) → { userId: 42, role: "admin" }

HTML

HyperText Markup Language — Structure & Semantics

HTML (HyperText Markup Language) is the foundational language of the web. It defines the structure and meaning of content using a system of nested elements — opening tags, content, and closing tags — that browsers parse into a tree called the DOM (Document Object Model). CSS then styles the DOM; JavaScript manipulates it.

Modern HTML5 introduced semantic elements (<header>, <nav>, <main>, <article>, <section>, <aside>, <footer>) that carry meaning rather than purely visual structure. This benefits SEO (search engines understand page hierarchy), accessibility (screen readers navigate landmarks), and maintainability (code is self-documenting).

Semantic Document Structure

A well-structured HTML document uses semantic elements to create meaningful landmarks. This allows search engine crawlers and assistive technologies to understand the layout without relying on visual cues.

<html lang="en">
  <head> ... </head>
  <body>
    <header> Logo + Nav </header>
    <main>
      <article>
        <section> Content </section>
      </article>
      <aside> Sidebar </aside>
    </main>
    <footer> Copyright </footer>
  </body>
</html>
Accessibility (a11y)

Accessibility ensures content is usable by people with visual, motor, or cognitive disabilities. Web Content Accessibility Guidelines (WCAG 2.1) define three conformance levels: A, AA, and AAA.

Key practices:

  • alt attributes on all images — describe the image for screen readers
  • aria-label / aria-describedby for context-free icons
  • aria-expanded, aria-hidden to communicate dynamic state
  • Every <input> paired with a <label for="...">
  • Logical heading hierarchy: only one <h1>, then <h2>…
  • Keyboard focus order matches visual layout

Render-Blocking Resources

The browser cannot render the page until it has downloaded and parsed all render-blocking CSS and JavaScript. Misplaced <script> tags freeze the HTML parser, causing a blank screen. Two attributes solve this:

  • defer — downloads in parallel, executes after DOM is fully parsed
  • async — downloads in parallel, executes immediately when ready (order not guaranteed)
/* ✓ Non-blocking — preferred */
<script src="app.js" defer></script>

/* ✓ Lazy-load off-screen images */
<img src="hero.jpg" loading="lazy" alt="...">

/* ✓ Preconnect to external origins */
<link rel="preconnect" href="https://fonts.googleapis.com">

CSS

Cascading Style Sheets — Layout, Styling & Animation

CSS (Cascading Style Sheets) controls the visual presentation of HTML. The word cascading refers to the specificity algorithm that determines which rule wins when multiple declarations target the same element. Rules cascade from the browser's defaults → author stylesheets → inline styles, with !important at the top of the chain.

CSS is governed by the Box Model: every element is a rectangular box composed of content → padding → border → margin. Understanding this model — and setting box-sizing: border-box globally — is foundational to predictable layouts. Modern CSS provides two primary layout systems: Flexbox (one-dimensional) and CSS Grid (two-dimensional).

Specificity Hierarchy

When two rules conflict, the browser applies the one with higher specificity. Specificity is calculated as a four-part score (inline, IDs, classes, elements).

1000style="..."Inline styles
0100#idID selector
0010.classClass / pseudo-class / attribute
0001elementType & pseudo-element
Rendering Pipeline & Performance

To achieve smooth 60fps animations, avoid properties that trigger Layout recalculation (reflow). Only transform and opacity are composited on the GPU — bypassing Layout and Paint entirely.

Responsive Design — Mobile-First Methodology

Mobile-First means writing base styles for small screens first, then progressively enhancing with min-width media queries. This produces leaner CSS and ensures core content is always accessible. 

/* Base — mobile (≤ 576px) */
.card { width: 100%; }

/* Tablet (≥ 768px) */
@media (min-width: 768px) {
  .card { width: 50%; }
}

/* Desktop (≥ 1200px) */
@media (min-width: 1200px) {
  .card { width: 33.33%; }
}

Flexbox — One-Dimensional Layout

Flexbox distributes space along a single axis (row or column). The flex container controls alignment; flex items can grow, shrink, and wrap. Essential properties: 

.container {
  display: flex;
  flex-direction: row;        /* axis */
  justify-content: space-between; /* main axis */
  align-items: center;        /* cross axis */
  flex-wrap: wrap;            /* allow wrapping */
}
.item { flex: 1 1 200px; }   /* grow shrink basis */
CSS Grid — Two-Dimensional Layout

CSS Grid is the first native 2D layout system. It defines explicit rows and columns, enabling complex page layouts without hacks. Items can span multiple cells.

.grid {
  display: grid;
  grid-template-columns:
    repeat(3, 1fr);           /* 3 equal columns */
  grid-template-rows:
    auto 1fr auto;
  gap: 16px;
}
.header { grid-column: 1 / -1; } /* full width */
CSS Variables & Custom Properties

CSS Custom Properties (variables) store reusable values — colours, spacing, typography — in the :root scope. They cascade like regular properties and can be overridden in component scopes, enabling theming without JavaScript.

/* Define tokens at root */
:root {
  --color-brand: #38bdf8;
  --space-md: 16px;
  --radius: 12px;
}

/* Consume anywhere */
.button {
  background: var(--color-brand);
  padding: var(--space-md);
  border-radius: var(--radius);
}

Bootstrap

CSS Framework — Grid, Components & Utilities

Bootstrap 5 is the world's most popular open-source CSS framework. It provides a mobile-first grid system, a comprehensive library of pre-styled UI components (buttons, modals, alerts, navbars), and a utility API inspired by atomic CSS — all with minimal JavaScript dependencies. Bootstrap accelerates development by abstracting low-level CSS into semantic class names.

At its core, Bootstrap's layout is built on CSS Flexbox organised into a 12-column grid with six responsive breakpoints. The Utility API (v5+) allows custom utilities to be generated from SCSS configuration, meaning teams can ship only what they use — reducing payload by up to 80% compared to importing the full bundle via CDN.

12-Column Responsive Grid

Bootstrap's grid works by nesting .col-* classes inside .row inside .container. Columns always add up to 12 per row. Responsive modifiers (-sm-, -md-, -lg-) override the column width at specific breakpoints. 

<div class="container">
  <div class="row">
    <div class="col-12 col-md-8 col-lg-6">
      /* 12 cols mobile, 8 tablet, 6 desktop */
    </div>
    <div class="col-12 col-md-4 col-lg-6">
      /* remaining columns */
    </div>
  </div>
</div>

Auto columns with .col divide available space equally. Use offset-md-2 to push columns right. Nest rows inside columns for complex layouts.

JavaScript

Dynamic Behaviour, DOM Manipulation & Async Programming

JavaScript (JS) is the only programming language natively executed by web browsers. It transforms static HTML/CSS pages into interactive applications by manipulating the DOM, handling user events, and communicating with servers asynchronously. Every modern browser ships a JavaScript Engine (V8 in Chrome/Node.js, SpiderMonkey in Firefox) that compiles JS to optimised machine code via Just-In-Time (JIT) compilation.

JS is single-threaded — it has one call stack and can only execute one task at a time. Long-running operations (network requests, file I/O) are handled by the browser's Web APIs, which run on separate threads. When complete, their callbacks enter the Task Queue (macro-tasks) or Microtask Queue (Promises). The Event Loop continuously checks whether the call stack is empty and pushes the next queued task to run.

The Event Loop

The Event Loop is the mechanism that lets JavaScript perform non-blocking I/O despite being single-threaded. It processes the Call Stack first, then drains all Microtasks (Promises), then takes one Macro-task (setTimeout, fetch callback) and repeats.

// Execution order demonstration
console.log("1 — synchronous");

setTimeout(() => {
  console.log("3 — macro-task");
}, 0);

Promise.resolve().then(() => {
  console.log("2 — microtask");
});

// Output: 1, 2, 3
Async/Await & Promises

Promises represent the eventual result of an async operation — either fulfilled or rejected. async/await is syntactic sugar over Promises that makes async code read like synchronous code.

Always wrap await calls in try/catch to handle network errors, timeout issues, and API errors gracefully without crashing the application. 

async function fetchUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(res.status);
    return await res.json();
  } catch (err) {
    console.error("Fetch failed:", err);
  }
}

DOM Manipulation

The Document Object Model is a tree-like, in-memory representation of the HTML page. JavaScript can query, create, modify, and delete any node. Modern code prefers querySelector over the older getElementById. 

// Select elements
const el = document.querySelector(".card");

// Read / write content
el.textContent = "Hello!";
el.classList.toggle("active");
el.setAttribute("aria-hidden", "true");

// Create & append
const p = document.createElement("p");
p.textContent = "New paragraph";
el.append(p);
Error Handling & Resilience

Robust applications handle errors gracefully. The three-part strategy:

  1. Prevent: validate input before sending
  2. Catch: wrap async operations in try/catch
  3. Recover: show meaningful UI feedback (not raw error objects)
async function loadData() {
  try {
    showSpinner();
    const data = await fetchUsers();
    renderList(data);
  } catch (err) {
    if (err.status === 404) showEmptyState();
    else showErrorToast(err.message);
  } finally {
    hideSpinner();
  }
}
